### **1. 시스템 콜이 무엇인지 설명해 주세요.**
A. 시스템 콜(System Call) 이란 애플리케이션에서 운영 체제로 서비스를 요청하기 위해 호출하는 인터페이스로,
애플리케이션은 직접 하드웨어 자원(CPU, 메모리 등)에 접근할 수 없으므로 시스템 콜을 통해 운영 체제에 간접적으로 요청을 전달한다.

- **우리가 사용하는 시스템 콜의 예시를 들어주세요.**
  1. 파일 처리 관련: 파일을 열거나 읽고 쓰는 작업
     - open(), read(), write(), close()
  2. 프로세스 관리
     - fork(): 새로운 프로세스 생성.
     - exec(): 현재 프로세스를 새로운 프로그램으로 대체.
     - exit(): 프로세스 종료.
  3. 메모리 관리
     - mmap(): 메모리 매핑.
     - brk(): 힙 메모리 크기 조정.
  4. 네트워크 통신
     - socket(): 소켓 생성.
     - connect(): 소켓 연결.
     - recv(), send(): 데이터 송수신.

- **시스템 콜이, 운영체제에서 어떤 과정으로 실행되는지 설명해 주세요.**
  1. 유저 모드 → 커널 모드 전환
     - 애플리케이션에서 시스템 콜 라이브러리 호출 (read() 등).
     - 시스템 콜 번호와 매개변수가 레지스터 또는 스택에 저장됨.
     - CPU가 특권 명령 (Trap Instruction) 실행 후 커널 모드로 전환.
  2. 운영 체제의 처리
     - 커널이 시스템 콜 번호를 확인하여 해당 서비스 함수 실행.
     - 하드웨어 자원 접근, 작업 수행.
  3. 결과 반환
     - 처리 결과를 사용자 애플리케이션으로 반환.
     - CPU 제어권을 유저 모드로 전환.

- **시스템 콜의 유형에 대해 설명해 주세요.**
  - 프로세스 제어: 프로세스 생성, 종료, 속성 변경, 대기 (fork(), exec(), exit()).
  - 파일 관리: 파일 읽기, 쓰기, 삭제, 속성 변경 (open(), read(), write(), unlink()).
  - 디바이스 관리: 하드웨어 장치 제어 (ioctl(), read(), write()).
  - 통신: 프로세스 간 데이터 송수신 (pipe(), socket(), send(), recv()).
  - 정보 유지: 시스템 시간 확인, 환경 변수 조회 (gettimeofday(), getpid()).

- **운영체제의 Dual Mode 에 대해 설명해 주세요.**
A. 운영 체제는 Dual Mode 를 통해 유저 모드(User Mode)와 커널 모드(Kernel Mode)를 구분한다.
  - 유저 모드: 제한된 권한, 직접 하드웨어 접근 불가.
  - 커널 모드: 모든 권한을 가짐, 하드웨어 제어 가능.
- **왜 유저모드와 커널모드를 구분해야 하나요?**
  1. 보안: 애플리케이션이 직접 하드웨어를 제어하면 시스템이 불안정해질 수 있기 때문에, 커널 모드를 통해 하드웨어 접근을 제어함으로써 안정성을 보장한다.
  2. 자원 관리: 운영 체제가 자원을 관리하여 충돌 방지.
  3. 시스템 안정성: 프로세스가 커널 모드로 전환하지 않고는 시스템 자원을 제어할 수 없으므로, 잘못된 접근으로 인한 시스템 손상을 방지.

- **서로 다른 시스템 콜을 어떻게 구분할 수 있을까요?**
  1. 시스템 콜 번호: 각 시스템 콜은 고유한 번호를 가지며, 애플리케이션은 이 번호를 사용하여 원하는 시스템 콜 요청.
  2. 매개변수 확인: 시스템 콜 함수는 입력 매개변수와 반환값으로 구분 가능.
  3. 시스템 호출 테이블: 운영 체제는 시스템 호출 테이블(System Call Table)을 사용하여 번호에 맞는 시스템 콜 핸들러를 실행.

### **16. Thrashing 이란 무엇인가요?**
A. Thrashing(스레싱) 이란 운영체제에서 과도한 페이지 교체로 인해 시스템 성능이 급격히 저하되는 상태를 말한다.
주로 가상메모리를 사용하는 환경에서 발생하며, 메모리에 필요한 페이지가 부족해 디스크의 페이지 파일을 반복적으로 읽고 쓰는 상황이나
CPU 가 실제 작업보다 페이지 교체에 더 많은 시간을 소비할 때 발생한다.

- **Thrashing 발생 시, 어떻게 완화할 수 있을까요?**
  1. 멀티프로그래밍 수준 조정: 실행 중인 프로세스 수를 줄여서 각 프로세스가 충분한 메모리를 사용할 수 있도록 조정.
  2. Working Set 정책 사용
     - 프로세스가 자주 사용하는 페이지 집합(Working Set)을 유지하도록 관리.
     - Working Set 을 기반으로 적절한 페이지 교체 수행.
  3. 페이지 교체 알고리즘 최적화 
     - LRU(Least Recently Used), Clock Algorithm 등 효율적인 알고리즘 사용.
     - 잘못된 페이지 교체로 인해 필요한 페이지가 계속 교체되지 않도록 예방.
  4. 메모리 크기 확장: 물리적 메모리(RAM)를 증설하여 페이지 폴트를 줄임.
  5. 프로세스 우선순위 조정: 중요하지 않은 프로세스를 일시 정지하거나 종료하여 중요한 프로세스가 더 많은 메모리를 사용하도록 설정.
  6. Pre-Paging (페이지 미리 적재): 예상되는 페이지를 미리 메모리에 로드하여 페이지 폴트를 줄임.
